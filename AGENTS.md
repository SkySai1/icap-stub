# Product overview
ICAP Stub Server — тестовый ICAP-сервер, который принимает ICAP-запросы и возвращает ответы по заранее заданным параметрам.
Сервер используется как заглушка для тестирования интеграций и сценариев обработки ICAP.

# Repository structure
- `./app` — основной код продукта
- `./tests` — автоматические тесты
- `./docs` — документация по модулям и архитектуре
- `./venv` - виртуальное окружения python3

# Tech stack
- Python 3.12
- Документация ведётся в Markdown
- Конфигурация сервера задаётся через `config.ini`
- Работы с проектам должна происходить через venv

# Development rules

## Architecture
- Каждая крупная зона ответственности должна быть вынесена в отдельный модуль.
- Не смешивать в одном модуле:
  - обработку сетевых соединений,
  - парсинг конфигурации,
  - бизнес-логику формирования ICAP-ответов.
- Логику, зависящую от конкретного порта, выносить в отдельные обработчики или стратегии.

## Code style
- Имена функций, классов и переменных должны быть понятными и отражать назначение.
- Избегать неочевидных сокращений.
- Для каждой публичной функции и каждого класса писать docstring.
- Код должен быть разбит на небольшие функции с одной зоной ответственности.

## Documentation
- Для каждого основного модуля должен быть создан файл документации в `./docs`.
- Документация должна содержать:
  - назначение модуля,
  - входы,
  - выходы,
  - условия и основные ветки логики,
  - диаграмму работы в формате Mermaid.

# Functional requirements
- Сервер должен принимать входящие ICAP-запросы.
- Сервер должен поддерживать одновременную работу на нескольких портах.
- Для каждого порта должна поддерживаться отдельная логика ответа.
- Параметры поведения сервера должны задаваться через `config.ini`.
- Минимальный набор параметров:
  - IP-адрес прослушивания,
  - порт,
  - время отклика,
  - код ответа.

# Non-functional requirements
- Код должен быть пригоден для модульного тестирования.
- Архитектура должна позволять добавлять новые сценарии ответа без переписывания существующих модулей.
- Веб-интерфейс не нужен, только CLI формат управления
- Проект должен быть docker ориентированным: иметь Dockerfile и compose.yaml

# Commands
- Run app: `python -m app.main`
- Run tests: `pytest tests/`
- Lint: `ruff check .`
- Format: `ruff format .`

# Definition of done
Задача считается завершённой, если:
- код проходит тесты;
- код проходит линтер;
- добавлена или обновлена документация в `./docs`;
- новая функциональность покрыта тестами.
